# Website-Technologies-Scraper
This project is a lightweight technology detection tool similar to Wappalyzer, that analyzes websites and tries to identify the technologies they use. The goal was to build a simple rule-based system that can inspect a list of domains and detect things such as CMS platforms, analytics tools, frontend frameworks, infrastructure providers, and other common website technologies.

How it works
The project takes a list of domains from a .parquet file and processes them one by one. For each domain, it sends a request to the website, parses the returned HTML, and extracts useful signals such as script URLs, stylesheet links, meta tags, headers, cookies, inline scripts, and DOM markers. These signals are then compared against a database of predefined rules stored in tech_rules.json.
When a technology matches one or more rules, it is added to the result together with a confidence score(0.6 for 1 proof, 0.8 for 2 proofs, 0.95 for 3 or more proofs) and a short proof showing why it was detected.
The detection engine is based on pattern matching. Each technology has a set of rules that may look for keywords in the HTML, script sources, link URLs, meta tags, headers, cookies, or regular expressions. The more signals are matched for a given technology, the higher the confidence score becomes.
The scraper uses multithreading so that multiple websites can be processed in parallel. This makes the solution much faster than checking domains one by one. During execution, the current progress is printed in the console so it is easy to see how many domains have already been processed. With the current implementation, the tool was able to detect around 80 unique technologies.

Limitations
This solution is based on visible frontend signals, so it cannot guarantee perfect accuracy. Some websites hide or minimize the technology traces that they expose, and some technologies are only visible after deeper interaction or full browser rendering. Another limitation is that the current solution does not perform deep crawling across the entire site, so it may miss technologies that only appear on internal pages.
If I had more time, I would improve the project by using a headless browser for richer rendering, by crawling more internal pages per domain, and by expanding the technology rules database even further. I would also work on reducing false positives and making the confidence scoring more precise.
